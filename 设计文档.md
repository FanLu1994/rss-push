# RSS -> AI 摘要 -> Telegram 推送系统设计文档

## 1. 目标与范围

### 1.1 目标
- 定时检查多个 RSS 订阅源。
- 识别“新文章”（避免重复处理）。
- 将新文章交给 AI 分析，输出简要介绍与可读摘要。
- 汇总后通过 Telegram Bot 发送到指定聊天。

### 1.2 非目标（首版）
- 不做 Web 管理后台。
- 不做复杂推荐算法。
- 不做全文存档与搜索。

## 2. 关键需求

### 2.1 功能需求
- 支持配置多个 RSS URL。
- 支持 GitHub Actions 定时执行（例如每小时一次）。
- 支持增量处理：仅处理未推送过的文章。
- 每次执行生成一条或多条 Telegram 消息。
- AI 输出格式统一（标题、来源、摘要、价值点、链接）。

### 2.2 非功能需求
- 幂等：重复触发不会重复推送同一文章。
- 可观测：执行日志可追踪失败原因。
- 成本可控：限制每次处理文章数量与模型 token。
- 可维护：配置与代码解耦。

## 3. 总体架构

```text
GitHub Actions (schedule/workflow_dispatch)
  -> 拉取 RSS 列表配置
  -> 抓取并解析 Feed
  -> 新文章判定（基于去重存储）
  -> 内容清洗（标题/摘要/正文片段）
  -> 调用 AI 生成简介
  -> 汇总消息模板渲染
  -> Telegram Bot API 发送
  -> 更新去重状态（已处理 article_id）
```

## 4. 模块设计

### 4.1 Feed 抓取模块
- 输入：`feeds.yaml`（或 `feeds.json`）中的 RSS 列表。
- 处理：
  - 拉取 XML。
  - 解析 `title/link/pubDate/guid/description/content`。
  - 标准化时间格式为 UTC。
- 输出：统一 Article 对象数组。

### 4.2 去重与状态模块
- 方案 A（推荐首版）：用仓库文件 `state/processed.json` 记录已处理 `article_id`。
- `article_id` 生成规则（优先级）：
  1. `guid`
  2. `link`
  3. `hash(title + source + pubDate)`
- 每次成功发送后更新状态文件并 commit 回仓库（或 artifacts/cache 方案）。

### 4.3 AI 分析模块
- 输入：文章标题 + 摘要（必要时正文前 N 字）。
- 输出结构（JSON）：
  - `brief`: 1~2 句简介
  - `highlights`: 2~4 个要点
  - `why_it_matters`: 为什么值得关注
  - `tags`: 主题标签
- 约束：
  - 单篇 token 上限。
  - 超时与失败重试（1~2 次）。
  - 失败时降级：直接使用原始摘要。

### 4.4 汇总与消息渲染模块
- 按执行批次聚合新文章。
- 生成 Telegram Markdown 消息，包含：
  - 批次标题（时间窗口）
  - 每篇文章：标题、来源、AI 简介、链接
- 超长消息拆分发送（Telegram 长度限制）。

### 4.5 Telegram 推送模块
- 使用 Bot API `sendMessage`。
- 配置：
  - `TELEGRAM_BOT_TOKEN`
  - `TELEGRAM_CHAT_ID`
- 错误处理：
  - 429 限流退避重试。
  - 网络异常重试。

## 5. 数据模型（建议）

### 5.1 `feeds.yaml`
```yaml
feeds:
  - name: "Hacker News"
    url: "https://hnrss.org/frontpage"
    enabled: true
  - name: "TechCrunch AI"
    url: "https://techcrunch.com/category/artificial-intelligence/feed/"
    enabled: true
```

### 5.2 运行时 Article（逻辑模型）
```json
{
  "id": "stable-id",
  "source": "TechCrunch AI",
  "title": "Article title",
  "link": "https://...",
  "published_at": "2026-03-22T09:00:00Z",
  "summary_raw": "...",
  "content_raw": "..."
}
```

### 5.3 `state/processed.json`
```json
{
  "updated_at": "2026-03-22T10:00:00Z",
  "items": {
    "stable-id-1": "2026-03-22T10:00:00Z",
    "stable-id-2": "2026-03-22T10:00:00Z"
  }

```

## 6. GitHub Actions 设计

### 6.1 触发策略
- `schedule`: `cron: "0 * * * *"`（每小时）。
- `workflow_dispatch`: 手动触发调试。

### 6.2 Job 步骤
1. Checkout 代码。
2. 安装依赖（Node.js/Python 二选一）。
3. 执行主脚本（抓取 -> 分析 -> 推送）。
4. 若状态文件变更，提交回仓库。

### 6.3 Secrets 与变量
- `OPENAI_API_KEY`（或你选用的 LLM key）
- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`

## 7. Prompt 与输出规范

### 7.1 单篇分析 Prompt（示意）
- 输入：标题、来源、发布时间、摘要。
- 指令：
  - 生成中文简介（40~80 字）。
  - 生成 2~4 条关键点。
  - 给出 1 条“为什么重要”。
  - 输出 JSON，禁止额外文本。

### 7.2 质量控制
- JSON 解析失败时自动重试一次。
- 仍失败则使用降级模板：
  - `brief = summary_raw 截断`
  - `highlights = []`

## 8. 失败处理与可靠性

- RSS 拉取失败：跳过该源并记录 warning，不影响其他源。
- AI 调用失败：单篇降级，不阻塞整批。
- Telegram 发送失败：重试，最终失败时 job 标记失败（便于告警）。
- 去重状态写入失败：job 失败，防止下次重复推送大量历史。

## 9. 成本与性能控制

- 每次执行最多处理 `N` 篇新文章（如 20）。
- 文章正文截断（例如最多 1200 字符输入 AI）。
- 仅在“新文章”上调用 AI，避免重复消耗。
- 可以按来源设置优先级与配额。

## 10. 安全设计

- 所有密钥仅存 GitHub Secrets，不入库。
- 日志中不打印密钥与完整请求体。
- 仅允许机器人发送到指定 Chat ID（白名单）。

## 11. 里程碑

### M1（MVP，1~2 天）
- RSS 抓取 + 去重 + Telegram 原文推送（不加 AI）。

### M2（+1 天）
- 接入 AI 摘要与统一消息模板。

### M3（+1 天）
- 增加失败重试、降级、状态自清理（TTL）。

### M4（可选）
- 增加按主题分类、每日汇总周报、多语言输出。

## 12. 验收标准

- 定时任务按预期触发（可在 Actions 历史中看到）。
- 同一篇文章不会重复推送。
- 每条推送包含标题、简介和原文链接。
- 任一 RSS 源异常时，其它源仍正常推送。
- 在配置上限内，单次执行耗时与 token 成本可接受。

## 13. 推荐技术栈（首选 Node.js）

- 运行时：Node.js 20
- RSS 解析：`rss-parser`
- HTTP：`undici` 或 `axios`
- 配置校验：`zod`
- 日志：`pino`
- Telegram：直接调用 Bot API（减少依赖）

## 附录 A：主流程伪代码

```ts
main():
  feeds = loadFeeds()
  state = loadProcessedState()
  articles = fetchAndParseAll(feeds)
  fresh = filterNewArticles(articles, state)
  fresh = sortByPublishedAtDesc(fresh).slice(0, MAX_ITEMS)

  results = []
  for article in fresh:
    ai = summarizeWithAI(article) || fallbackSummary(article)
    results.push(renderItem(article, ai))

  if results not empty:
    messages = chunkTelegramMessages(renderBatch(results))
    sendAll(messages)
    markProcessed(state, fresh)
    saveState(state)
  else:
    log("no new articles")
```
